Gitかるたの簡単説明書
--------------------------------------------------------
1.git add
  ファイルやディレクトリをインデックスに登録
2.git commit
  ローカルリポジトリに変更を反映する
3.git commit -a
  修正 / 削除された全てのファイルについてローカルリポジトリに変更を反映する
  ただし、新しいファイルは無視される
4.git commit -amend
  コミット内容を修正する場合に利用するコマンド。
  わざわざ修正の繁栄に新規コミットするのではなく、最新のコミット内容を修正するとかで使う。
  コミットログを修正する場合などにも活用できる。
5.git branch fix/42
  "fix/42"という名前のブランチを作成する（現在ブランチから派生）
6.git checkout -b fix/42; git commit
  新たに「fix/42」という名前のブランチを作成し、HEADが「fix/42」に移動する。
  その後、コミットを実行する。
7.git checkout -b fix/42
  指定した名前のブランチを作成する(この場合、fix/42)。また、HEADが指定したブランチに移動する
8.git reset -hard master
  ブランチ作業を捨てて、ローカルのmasterを強制的にリモートのmasterに合わせる
9.git merge fix/42
  fix/42ブランチを現在ブランチにマージする
10.git revase -i A~E
  リビジョンAからEまでのコミット履歴を対話的に編集する。
  本コマンドを実行すると対象範囲のコミットに対して「どのように編集するか」の
  指示書がエディタで開かれる。指示書を編集して保存し、エディタを終了すると、
  指示書に基づく編集が実施される。
12.git checkout fix/42
   指定したファイル名のブランチを作成する。ただし、HEADは更新されない。
13.git rebase master
  現在のブランチが生えているポイントをmasterの最新のコミットに変更する
14.git merge --ff-only fix/42
  マージ対象のブランチとfast-forwardの関係にある場合、参照先の更新を行う（進める）
　fast-forwardとして扱えない場合は、何もしない（失敗する）
15.git marge --no-ff fix/42
  マージがfast-forward(マージ先に変更がない場合はマージコミットは必要なく、
  ブランチが指し示す場所だけが変更される)として扱える場合でも、マージコミットを生成する。
16.git fetch all
   全てのリモートの変更をローカルに反映する
17.git remote add; git fetch -all
  開発の途中で複数のリポジトリに分けて開発をしていた場合に
  分岐した情報も含めたすべてのリモートリポジトリをローカルに取得する。
  HEADの位置は変わらない。
18.git pull
  共有リポジトリのコミットをローカルリポジトリに「マージした上で」取り込む。
19.git pull --rebase
  共有リポジトリのコミットをローカルリポジトリに取り込む。
  「git fetch」 + 「git rebase」の動作
20.git push
   ローカルでの変更をリモートに反映する
   反映の際、マージが実施される。リモート / ローカルのmasterとHEADが一致した状態となる
21.git cherry pick 2
  特定のコミット内容のみを適用する
22.git init
  カレントディレクトリに.gitフォルダを生成し、Gitリポジトリとして扱えるようにする。
23.git clone
  共有リポジトリを複製してローカルリポジトリとして取得する。

ローカルブランチからの基本的な操作

--ローカルブランチでの作業
1. git chechout ブランチ
2．ファイルを編集
3．git add ファイル名
4. git commit
--ローカルマスターでの作業
5. git chechout master
6. git pull   <- ローカルマスターとリモートマスターを同期：更新が入ればconflictあり??? このへんの手順が怪しい
7. git marge umetani  <- conflictがある場合修正
8. git push  <- リモートにローカルの編集を反映
